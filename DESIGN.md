# HalfHalf 架构设计：从"自动排版"到"人机协作排版"

这份文档记录产品的分层架构和后端接口契约，供前后端对齐。它是**设计意图**的记录，
不是已完成状态——各层的完成情况见 README 的开发计划。

---

## 为什么是这个架构：机器和人各做各擅长的

做一份半开卷小抄，人做的是两种劳动：

1. **内容压缩（脑力活）**：判断什么重要、把叙述砍成极简碎片。→ AI 擅长（语言任务）。
2. **空间塞放（体力活）**：把碎片在纸上密堆、拼满。其中"二维密堆"是 AI 的死角——
   内容会随宽度重排导致块没有固定尺寸、机器没有"眼睛"实时测量、且本质是 NP 难 + 语义判断。

结论：**别让机器做二维拼图**。机器做压缩、初排、渲染、导出；把"拼图"这一步的判断留给人。

---

## 三层架构

```
原始资料
  │
  ▼ ① AI 内容压缩        —— 自动化"总结"，最大价值。状态：未做
  │
  ▼ ② 自动默认排版        —— 懒人下限，一键打印。状态：已做（多栏引擎）
  │
  ▼ ③ 网格拖拽微调（可选） —— 重度用户上限，做到样张级密度。状态：未做，主要前端
  │
  ▼ 导出 PDF
```

三层解耦，可分别开发/上线：

- **只用 ②**：用户不想动手，自动排版直接打印，效果"勉强能用"。
- **② → ③**：愿意花几分钟的用户手动拼到完美。
- **① 叠加在最前**：让进入 ②③ 的内容本身就够碎，默认版和手工版都更密。

**关键认知**：正因为多数用户（考前熬夜的学生）不会动手拼图，**② 自动默认版的质量反而更重要**——它是大多数人实际拿到的东西。③ 是上限，② 是下限，都要有。

---

## ② 与 ③ 的统一：块 + 模板驱动的自动拼装

②（自动默认版）和 ③（手动编辑）**共享同一套数据模型**：内容块 + 网格位置。
② 不是另一条渲染管线，而是"程序按模板自动摆好块"；③ 就是在 ② 的结果上继续改。
**用户打开编辑器时永远不面对空白画布**——起点就是自动拼装的结果。

```
Markdown → 分块 → 按模板贪心拼装 → 自动默认版（可直接导出）
                                     │
                     用户想改 → 编辑器加载同一份排布，拖拽微调 → 导出
```

### 模板 = 拼装策略 + 风格预设（不是固定挖坑的版式）

固定槽位的模板对任意内容太脆（8 个坑遇到 11 个块就崩）。这里的模板不锁死块数量，
只描述"怎么拼"：

```ts
interface LayoutTemplate {
  id: string;
  gridColumns: number;      // 网格粒度，如 12
  gap: number;              // 块间距 mm
  packing: 'column-flow'    // 竖栏流：按阅读顺序从左往右填栏（似报纸，顺序好）
         | 'size-desc';     // 大块优先：先摆大块小块填缝（最密，顺序让位于密度）
  keepOrder: boolean;       // 是否严格保持阅读顺序
  style: {
    blockBorder: boolean;   // 块框线（手工样张的"框感"）
    headingBanner: boolean; // 一级标题通栏横幅
    density: Density;
  };
}
```

起步建议三个模板：`dense`（紧凑无框，密度优先）、`boxed`（样张风带框线）、
`airy`（宽松，可读性优先）。

### 为什么贪心拼装可行（与"二维密堆是 NP 难"不矛盾）

不可行的是"**最优**无缝密堆"；这里只做"**不错的**贪心密堆"：
每块先量出尺寸 → 按模板策略逐个找最靠前的可放位置 → 放不下换行。
贪心必然留小缝隙、不完美——这正是留给 ③ 用户手调的部分。机器 70-80%，人补最后一公里。

现有的"连续多栏流"引擎（`/api/optimize`）继续保留，作为纯文字密集内容的快速通道；
块模型不适合的场景（如整篇就是一篇连贯长文）仍走连续流。

## ③ 网格拖拽微调：交互模型

- **画布 = A4 页面切成网格**（如横向 12 格 × 纵向若干行）。
- **内容块 = 卡片**，每张占据整数个格子（1×1 到 N×M），可拖动、可缩放，网格吸附。
- **块尺寸变化时，内容自适应**：
  - 文字：重排（换行适应新宽度）+ 自动调字号填满盒子。
  - 公式/表格：不能重排，整体等比缩放（CSS transform）填满。
  - 图片/图表：纯几何缩放填满。
- **自适应实时性**：编辑器跑在浏览器里，浏览器测量文字尺寸是瞬时的，所以自适应字号
  **在前端实时算**，拖动即时响应，不需要后端往返。后端引擎只在**最终导出 PDF** 时用一次（高保真）。
- **护栏——最小可读字号**：盒子不能被拖到比"内容在最小字号（如 6pt）下所需空间"还小；
  到底就拖不动（吸附阻力 / 硬下限），防止把内容压成不可读。
- **块尺寸默认怎么来**：先纯自动（按内容长度/类型启发式给默认大小），
  后续可选让作者对少数重点块加轻量标记（如 `!重点`）来加权。不做固定模板（对任意内容太脆）。

## 后端接口契约（支撑 ③，供前端独立开发）

后端不懂拖拽，只当"渲染器 + 导出器"。前端负责画布交互和实时预览。两者靠这两个接口对接。

> 以下为**设计草案**，实现前可再调整；最终以 `packages/server/API.md` 和 `types/index.ts` 为准。

### `POST /api/layout/auto`（分块 + 按模板自动拼装）

把 Markdown 切成卡片，按指定模板贪心拼装出每张卡片的网格位置/大小。
返回结果既是 ② 的自动默认版（可直接送 export 出 PDF），也是 ③ 编辑器的初始画布。

请求（草案）：
```ts
interface AutoLayoutRequest {
  markdown: string;
  targetPages: number;
  paperSize?: PaperSize;
  orientation?: ResolvedOrientation;
  template?: string;      // 模板 id：'dense' | 'boxed' | 'airy'，默认 'dense'
}
```

响应（草案）：
```ts
interface Block {
  id: string;
  markdown: string;        // 该卡片的原始 Markdown 片段
  grid: { col: number; row: number; colSpan: number; rowSpan: number };
}
interface AutoLayoutResponse {
  blocks: Block[];
  gridColumns: number;
}
```

### `POST /api/layout/export`（按排布导出 PDF）

前端把用户拼好的排布（每张卡片的最终网格位置/大小）发回来，后端用 Playwright 高保真渲染成 PDF。

请求（草案）：
```ts
interface ExportLayoutRequest {
  paperSize?: PaperSize;
  orientation?: ResolvedOrientation;
  gridColumns: number;
  blocks: Block[];         // 含用户调整后的 grid 位置/大小
  // 每块字号可为具体值，或 'fit'（导出时按盒子自适应，复用字号搜索能力）
  fontSizeMode?: 'fit' | number;
}
```

响应：`200` + PDF 二进制流（同 `/api/render`，`Content-Disposition: inline` + `X-Page-Count`）。

## 与现有引擎的关系

- **两种渲染模型并存**：连续多栏流（`/api/optimize`，适合连贯长文）和卡片网格
  （`/api/layout/*`，模板拼装 + 可编辑）。② 的默认版走卡片网格；编辑（③）只对卡片网格开放。
- **字号自适应**复用现有的二分搜索字号能力，只是把目标从"塞进 N 页"换成"填满盒子"。
- **导出**复用 `render-pdf.ts` 的 Playwright + KaTeX + Shiki + Mermaid 渲染管线。
