# 排版规则规范（RULES.md）

> 这份文档定义 HalfHalf 自动排版的**决策规则**：给定一份 Markdown，如何推导出排版参数
> （`minScale` / `density` / `widthTiers` / `strategy` / `gutterMm` / `maxAspect` 及回填、驱逐开关）。
>
> 它是 `scene-presets.ts`（四个场景预设 + 优先级链推荐器）的**继任设计**。现有代码是它的
> 粗糙前身，本文档指明目标形态；代码按此逐步收敛。**规则想错了改这份文档便宜，改代码贵。**

---

## 0. 为什么要两层

早期做法是「四个学科场景预设 + 优先级链推荐」。它有两个根本毛病：

- **优先级链首个匹配即赢**：一份「截图多 + 公式多」的半导体材料先命中 `visual`，公式就被
  `minScale 0.7` 静默缩小——冲突没有裁决，只有静默失败。
- **按学科名分类不封闭**：世界上的学科无穷多，四个筐永远装不下，新学科没有归宿。

解法是把决策拆成两层：

| 层 | 管什么 | 来源 | 能否被抄 |
|----|--------|------|---------|
| **力学层** | 内容里**能测出来**的物理事实：有多少刚性原子、多少柔性文字 | 确定性统计 | 算法可抄 |
| **学科层** | 力学层**测不出**的领域知识：顺序能不能打乱、哪类原子是核心、领域惯例 | 真实材料 + 用户反馈积累 | **积累抄不走** |

力学层保证**任何输入都有合理兜底**（新学科自动落到某个力学画像）；学科层是**护城河**
——它编码的是"数学推导顺序不能乱""政治课一问几面顺序无所谓"这类无法从文本可靠推断、
只能靠领域经验沉淀的事实（与 [README](./README.md) 「护城河」一节的判断一致）。

**关键约束（防止学科层变成后门）**：学科层只能做两件事——① 提供力学层测不出的**补充特征**
（顺序刚性、原子角色）；② 在力学层划定的**可行域内**给参数建议值。**学科层不得突破力学层
的硬约束**。这样两层永远不会打架：力学层保证下限，学科层锦上添花。

## 一、力学层

> **证据等级（下文各表通用记号）**——约束体系里"有据"和"拍脑袋"的东西混在一起，
> 不显式标出来，几个月后自己都分不清哪些数字能信、哪些随时该推翻：
>
> | 记号 | 等级 | 含义 |
> |------|------|------|
> | ● | 已校准 | 数值经真实材料/实测背书，**写得出出处** |
> | ◐ | 方向可信 | 机制方向有依据，**具体数字是拍的** |
> | ○ | 占位 | 纯假设，未经任何材料检验，随时可推翻 |
>
> ● 长什么样（有出处才算）：cram 密度档的行高/发丝线/标题行内化 ← 照四张真实小抄照片
> 逐项校准；图片档位 5% 吸附容差 ← 实测"差 1.3% 就跳大档、字号掉 6.5pt"修出来的；
> H1 的方向 ← 真实材料 test.md 的翻车现场（公式被缩到难读）。
> **◐ 升 ● 的唯一途径是真实材料**（见 §4.2）。

### 1.1 核心概念：刚性原子 vs 柔性内容

排版的一切困难都来自**刚性原子**——不能重新排版的东西：

| 类型 | 为什么刚性 | 放不下时只能 |
|------|-----------|-------------|
| 独立公式 `$$…$$` | KaTeX 渲染成固定二维盒子，无法折行 | 整体等比缩小 |
| 表格 | 列宽由内容决定 | 整体缩小 |
| 图片 | 像素尺寸固定 | 等比缩小 |
| 代码块 | 折行会破坏缩进语义 | 缩小或横向截断 |

**柔性内容**（正文、行内公式 `$…$`、列表）随容器宽度自由重排，任何宽度都排得好，
**不参与宽度决策**。

> 一句话：排版参数几乎全部由「这份材料有多少、多宽的刚性原子」决定。学科名只是这些
> 特征常见组合的俗称。

### 1.2 输入维度（每个都写清它在机械上影响什么）

**柔性量**
- `charCount`（剥掉公式/表格/代码/Markdown 标记后的正文字数）——决定"内容总量 vs 目标页数"
  的压力，进而决定 `density`。注意是**剥后口径**，比原始文件字数小得多。
- `inlineFormulaCount`、列表项数——计入柔性总量，但不单独触发任何硬约束。

**刚性原子（数量 + 宽度诉求）**——这是决策的主角
- 各类原子的**数量**和**密度**（个/千字）。
- 各类原子的**宽度诉求**：一个原子在参考字号下的自然宽度需要几格。公式塞不进 6 格就说明
  留 6 格档没意义。

**结构**
- `blockCount`、块长分布（中位数）——巨型块会钉死字号（见 [自适应细分](./README.md)），
  已由 `chunkMarkdown` 的 `maxBlockChars` 处理。

**顺序刚性（orderRigidity）——力学层测不出，占位待学科层填**
- 强：数学推导、代码教程（打乱=毁内容）。
- 弱：政治课"一问几面"、要点罗列（打乱几乎无代价）。
- 力学层默认**保守假定为强**（不敢乱动顺序）；靠学科层或用户声明放宽。

### 1.3 参数因果表

| 参数 | 由什么决定 | 机械原因 |
|------|-----------|---------|
| `minScale` | 刚性原子是否承载**核心信息** | 公式/代码是核心 → 缩小=毁内容 → minScale 高（0.85）；背诵课的表格缩小还能看 → minScale 低（0.5） |
| `density` | 柔性量 vs 目标页数 | 内容多、页数紧 → cram；宽松 → normal |
| `widthTiers` | **最宽**刚性原子的宽度诉求 | 公式普遍要 8 格 → 去掉 6 格窄档（留着只逼出缩放）；纯文字 → 保留窄档做马赛克 |
| `strategy` / `backfill` / `evict` | **顺序刚性** | 顺序强 → column-flow 且不回填（保证阅读顺序）；顺序弱 → 允许 size-desc / 回填 / 驱逐换密度 |
| `gutterMm` | 观感偏好 | **与内容无关**，纯审美。默认 1 格宽疏朗；cram 用 2mm |
| `maxAspect` | ⚠️ **无机械因果（缺口）** | 见 §四。目前是经验参数，实测非单调（os-large 在 2 和 1.2 各有局部最优） |

### 1.4 硬约束（先算，不可违反，冲突取最严）

每条硬约束把某个参数**钳到一个区间**。多条同时成立时，对每个参数取所有区间的**交集**
（下界取 max、上界取 min）。这是"图+公式混合""表格+公式混合"这类多刚性材料的正解——
不是选一个赢家，而是同时满足。

| 编号 | 触发条件 | 钳制 | 证据等级 |
|------|---------|------|---------|
| **H1 公式可读** | 独立公式密度 ≥ 5/千字 且 数量 ≥ 4 | `minScale ≥ 0.85`；`widthTiers` 去掉 <8 格档 | 方向 ●（test.md 真实翻车）· 阈值/钳制值 ◐ |
| **H2 图片保真** | 图片块 ≥ 2 或 占比 ≥ 15% | 图片档位吸附容差收紧；`density ≤ compact` 给图让位 | 方向 ◐ · 数值 ◐（仅档位吸附容差 5% 是 ●） |
| **H3 表格可读** | 表格数 ≥ 3 且 表是核心（学科层判定） | `minScale ≥ 0.7`；宽表允许升到 12 格 | ○（未实现，也无任何材料证明该怎么排） |
| **H4 代码不折行** | 代码块 ≥ 3 | `minScale ≥ 0.75`；代码块 `white-space` 不强制换行 | 方向 ◐（折行毁缩进是领域常识）· 数值 ◐（零真实编程材料） |

**交集为空时的失败行为（必须显式，不能静默）**：按优先级 H1 > H4 > H3 > H2 放宽最低优先级
的那条，并在 rule trace 里记一条**警告**（"图片保真与公式可读冲突，已优先保公式，图片可能偏小"）。
静默失败已修（见 §4.1(4)），但仍是"优先级链选一个赢家"的过渡形态，未做到取交集。
**这条优先级链本身是 ○**——从未有一份材料同时踩过三条线来检验这个排序对不对。

### 1.5 软偏好（在硬约束可行域内择优）

| 编号 | 条件 | 偏好 | 证据等级 |
|------|------|------|---------|
| **S1 密度优先** | 柔性量大、刚性原子少 | 往 cram 靠，gutter 收小 | 方向 ◐ · 大文本线 1500 字 ◐（cram 档本身是 ●） |
| **S2 允许乱序换密度** | 顺序刚性弱（学科层/用户声明） | 开启 size-desc / 回填 / 驱逐 | ○（顺序刚性无处声明，规则当前根本无法生效） |
| **S3 疏朗优先** | 目标页数宽松（内容远小于容量） | normal 密度，gutter 用满 1 格宽 | ◐（纯审美偏好，谈不上校准） |

## 二、学科层

### 2.1 学科规则能表达什么、不能表达什么

```
SubjectRule {
  id: string                       // 'calculus' | 'os' | 'politics' | ...
  aliases: string[]                // 关键词，用于"建议"识别（非强制）
  // ① 力学层测不出的补充特征：
  orderRigidity?: 'strong' | 'weak'
  atomRoles?: { table?: 'core'|'support', image?: ..., formula?: ... }
  // ② 可行域内的参数建议（不得突破力学层硬约束）：
  suggest?: Partial<{ gutterMm, maxAspect, ... }>
  evidence: string                 // 这条规则基于什么真实观察，必填，可追溯
}
```

**能**：补充顺序刚性、标注原子角色（政治课的表是 support、操作系统的对比表是 core）、
在可行域内建议观感参数。
**不能**：把 `minScale` 降到 H1 要求以下、关掉图片保真——**硬约束是地板，学科层只能在地板以上活动**。

### 2.2 学科识别

沿用「程序推荐 + 用户可改」：`aliases` 关键词命中给出**建议**，用户在前端可确认或改选。
识别错了顶多退回力学层兜底，不会出灾难。**永远不把学科识别当作硬输入。**

### 2.3 初始条目（本次会话已积累的真实观察，全部标注证据）

| 学科 | orderRigidity | 原子角色 | 关键观察（evidence） |
|------|--------------|---------|---------------------|
| 微积分 calculus | strong | formula=core | 真实材料 test.md：`\qquad` 在一个 `$$` 里并排 2~3 个公式，是宽度失控主因（泰勒公式块被推到 12 格卡死整页）；28 个独立公式 |
| 操作系统 os | strong | table=**core** | os-large：进程/页面置换对比表是考点本体，不能缩 |
| 半导体/电路 | strong | image=core, formula=core | 用户原话"图片优先保原尺寸"；图+公式混合的典型 |
| 政治 politics | **weak** | table=support | 用户原话"一个问题几个方面"，方面之间逻辑弱，可乱序（S2） |

### 2.4 维护流程

**数据存哪**：`scene-presets.ts` 旁边的 TS 常量（和 `SCENE_PRESETS` 同一模式）。
不搞 JSON/数据库——条目封顶 20~30 条，是**配置不是数据**；TS 能强制 `evidence` 必填，
JSON 还得自己写校验。等真有外部贡献者、或条目上百时再谈迁移。

**证据分两类，只有第二类是护城河**：

| 类型 | 例子 | 是护城河吗 |
|------|------|-----------|
| 领域常识 | "数学推导顺序不能打乱" | ❌ 学过的人都能写 |
| **经验观察** | "微积分资料爱用 `\qquad` 在一个 `$$` 里并排 3 个公式，是宽度失控主因" | ✅ **必须见过真材料** |

> 自查：§2.3 的四条初始条目里，**只有微积分那条属于第二类**（来自 test.md）。
> 真实进度比条目数看起来的少。

**一条规则的生命周期**（锚定材料，不是锚定想法）：

```
① 拿到真实材料（朋友的课，跨专业）
② 跑一遍，看输出哪里不对
③ 守门问题：「不知道这是什么课，这个问题还能被发现吗？」
     能  → 力学层的活（补统计/调阈值），禁止碰学科层
     不能 → 才轮到学科层
④ 材料脱敏后存成 fixture + 规则带 evidence，一起提交
⑤ 判例表加一行：没有 fixture 能证明它有用的规则 = 迷信
```

**第③步是纪律核心**。学科名是个筐、什么都能往里扔——不设这道闸，学科层几个月后就会
变成一堆"我觉得数学课该这样"的意见集合，那不是护城河是负债。
反例：`\qquad` 并排公式导致宽度失控——**这条不该进学科层**，因为力学层测得出
（"某个 `$$` 自然宽度 > 12 格"）。学科层只记录测不出的（顺序刚性、原子角色）。

**防退化三条硬规矩**：
1. `evidence` 必填且指到具体材料——写不出"哪份材料、什么现象"就是猜的。
2. 每条规则必须能被证伪——答不上"没有它会怎样"就别加。
3. 规则与 fixture 同生共死——删材料就得复查规则。

**朋友材料的入库边界（材料主要来源，必须先立规矩）**：
- **先问再用**：入库前取得明确同意，说清"会进开源仓库、公开可见"。
- **脱敏**：删掉姓名、学号、班级、联系方式。`samples/` 里那四张照片就含真实姓名+学号
  与小红书号，已在 `7.16.1` 提交进仓库——**推远程前应处理**（见 README 的口碑/信任论述）。
- **学术诚信**：只收"自己整理的知识点总结"，排除历年真题/答案/未公开课件原文
  （与 README「学生内容社区」一节的红线一致）。
- **来源标注**：fixture 顶部注释写清来源与授权状态（真材料 / 合成 ✳️）。

### 2.5 反馈回路与隐私边界

- **信号**：用户把自动推荐改成别的，就是一次"这个画像判错了"的反馈。
- **隐私红线**：优先靠**开发者自行收集真实样例** + **用户主动提交的反馈**，
  **不默认采集用户内容**。README「学生口碑/信任」是核心资产，一次"偷偷上传我的复习资料"
  级别的事件会直接透支它。宁可积累慢，不可越界。

## 三、自动模式决策流程

```
1. 算力学特征           analyzeContent（需修：见 §四缺陷）
2. 叠加学科补充特征      若用户声明/识别出学科，取 orderRigidity、atomRoles
3. 算硬约束 → 可行域    H1~H4 逐条钳制，取交集；交集为空按 §1.4 失败行为放宽 + 警告
4. 软偏好 + 学科建议择优 在可行域内应用 S1~S3 与学科 suggest
5. 输出参数 + rule trace 每条触发的规则记一条账，前端"推荐理由"由 trace 生成
6. 用户可强制预设覆盖    预设降级为"常见组合的命名快捷方式"，规则引擎才是真身
```

**rule trace 取代现在手写的 reason 字符串**：现在 `recommendScene` 里每个分支手拼一句
中文理由，与实际生效的参数脱节。trace 是"实际触发了哪些规则"的忠实记账，理由由它派生，
不会说谎。

### 4.2 判例覆盖：八画像已全部有判例，但**只有一份是真材料**

2026-07-15 补齐四个合成判例（code-heavy / image-formula / table-heavy / politics），
八画像全部可跑。但必须清醒：**11 个 fixture 里只有 test.md 是真实课程材料**（用户提供），
其余全是合成的。

真实材料 test.md 已经证明过合成判例的危险：我的复刻 calc-monthly 把公式数低估一半
（13 vs 28），基于它做的调参全部偏松。

**因此本仓库的判例分两类，用途不可混淆**：

| 类型 | 用途 | 能不能用合成的 |
|------|------|--------------|
| **行为判例** | 锁死确定性逻辑（"5 个代码块 → H4"） | ✅ 能，合成足够 |
| **阈值判例** | 校准数值（"代码块 ≥4 才算密集"） | ❌ **必须真材料** |

现有合成判例全部只承担第一类职责。**§1.4 的所有阈值仍是待校准状态**，等真实材料到位。

换成证据等级（§一 图例）的说法：**◐ 升 ● 的唯一途径就是阈值判例，即真实材料**。
每拿到一份新学科的真实总结，预期能把 1~2 个数字从 ◐ 变 ●——这也是"找朋友要跨专业
真材料"排在待办首位的原因。

### 4.3 无机械因果的参数

`maxAspect` 目前是纯经验值（证据等级 ○），且实测非单调（os-large 在 maxAspect=2 给 11pt、=1.2 反给 11.5pt，
因 12 格卡两块正好铺满一行、接缝更少）。规范暂无法给出它的因果规则，标记为**待研究**。

### 4.4 顺序刚性无法自动测

启发式（连接词密度、"一二三"并列 vs "步骤1/2"）都不可靠。**方案：力学层默认保守（顺序强），
由学科层或用户在前端显式声明放宽。** 不赌启发式。

### 4.5 阈值悬崖
